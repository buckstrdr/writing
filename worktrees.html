How I Give Every AI Agent Its Own Git Worktree (So They Can't Trash Each Other's Work)

Two agents, one repo, one very bad afternoon. The unglamorous git trick that fixed it for good.

buckstrdr · 5 min read

I once had two agents building two different features in the same repo at the same time. Very efficient, I thought, very clever of me.

Then agent A, halfway through editing three files, decided it needed a different branch to do its job — ran a git checkout — and yanked the floor out from under agent B, who was mid-edit. What I got back was a single working directory containing half of each feature, mashed together, that compiled perfectly and worked not at all. It took me longer to unpick than it would've taken to just build both myself.

That's what parallel agents in one working directory get you. And the fix isn't sterner prompts or telling them to be careful — they don't listen, same as the rest of us. The fix is giving each one its own git worktree. It's the least glamorous trick in my whole setup, and quietly the most load-bearing.

What a worktree actually is

A worktree is just a second working directory hanging off the same repo. Same history, same underlying .git, but checked out to its own branch, in its own folder. One command and you've got a complete, separate copy of the code, pinned to its own branch, sharing the repo's guts.

It's cheap, because it isn't a re-clone — the object store is shared. You just get another window into the same project that can't see the other windows. Which, after my mashed-together afternoon, was exactly what I wanted.

How I wire it into the team

Now, when a worker gets a job, it gets spun up in its own fresh worktree, on its own branch. It builds in there and it physically cannot touch the other agents' folders. "Stay in your lane" stops being a polite request an agent forgets thirty seconds later, and becomes a wall the filesystem enforces whether it likes it or not.

When it's done, the flow is boring and predictable — which is the whole point. Give the agent a sandbox, let it work, review the branch, merge it, bin the sandbox. Three agents, three sealed sandboxes, three clean branches to merge — instead of one contaminated directory I have to perform surgery on.

Why it beats the obvious alternatives

The instinct is to just let everyone work in one folder and switch branches as needed. That's exactly what bit me — one git checkout and someone's mid-flight work is on the wrong branch or stashed into oblivion. Worktrees mean nobody ever switches the branch out from under anyone, because each agent's branch just sits there in its own directory, untouched.

The other instinct is to clone the whole repo three times. That works, but the clones drift into separate little islands and it's heavier on disk. Worktrees share one object store, so they're lighter and every branch stays visible from every worktree. One repo, several hands, no collisions.

The honest gotchas — because there are a few

You can't check out the same branch in two worktrees. Git flatly refuses. Felt annoying the first time; it's actually a gift — it's git stopping two agents editing the same branch by accident, which is precisely the kind of thing my agents would otherwise do with great enthusiasm.

Cleanup needs discipline. Stale worktrees pile up if you're sloppy. And when an agent dies messily mid-job — which happens — you'll occasionally need git worktree remove --force, which is exactly as much fun as it sounds.

Disk and dependencies are real. Each worktree is a full checkout, and things like node_modules or a Python venv don't magically share. For most repos it's a non-issue. For a monster monorepo, budget for it.

The payoff

Three agents, three features, one repo, all at once — each sealed in its own sandbox, none able to lay a finger on the others. When they finish, I'm merging three clean branches, not doing forensics on a working directory two agents fought over in the dark.

It's dull plumbing. But dull plumbing is usually the exact thing standing between a slick demo and something you'd trust with real work while you're asleep — which, after that afternoon, is the only kind of trust I care about.
