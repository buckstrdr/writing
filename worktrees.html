How I Give Every AI Agent Its Own Git Worktree (So They Can't Trash Each Other's Work)

The unglamorous git trick that lets a handful of AI agents build in the same repo at the same time — without turning it into a merge horror show.

buckstrdr · 5 min read

In my last piece I said each agent in my little AI team "owns its own files" so they don't overwrite each other. A reader called me on it, fairly: "OK, but they're all working in the same repo — how does that not just explode?"

Good question. And the honest answer is that "owns its own files" as a *rule* isn't enough. Rules get forgotten — especially by an eager agent that's decided the quickest way to do its job is to switch branches out from under everyone else. You don't want a polite agreement. You want a wall.

The wall is git worktrees. It's not clever, it's not new, and hardly anyone uses it. But it's the single thing that turned "a few agents flailing in one folder" into actual parallel work.

The problem: one repo, one working directory, several agents

Picture two agents in the same checkout. Agent A is halfway through editing three files. Agent B decides it needs a different branch to do its job, so it runs git checkout other-branch — and just like that, the floor vanishes under Agent A. Its half-done work is now sat on the wrong branch, or stashed, or gone.

Or they both just… edit. And you come back to a working directory that's a Frankenstein of two half-finished features, which of course still compiles, so you don't notice until something's subtly broken in production. The worst kind of broken, as I keep banging on about.

The root of it: in one directory, you can only be on one branch at a time. Parallel agents and a single shared checkout are fundamentally at war.

What a worktree actually is

A git worktree is just a second working directory hanging off the same repo. Same history, same underlying .git object store — but checked out to its own branch, in its own folder.

git worktree add ../work/auth -b auth-feature

Run that and you've got a complete, separate copy of the code sitting in ../work/auth, on a branch called auth-feature, sharing the repo's guts. It's cheap, because it's not a re-clone — the object database is shared. You just get another window into the same project, pinned to its own branch.

How I wire it into the team

When the orchestrator hands a job to a worker, it spins that worker up in its own fresh worktree, on its own branch. The agent builds in there and it physically cannot see or touch the other agents' folders. "Stay in your lane" stops being something an agent has to remember and becomes something the filesystem enforces.

When the worker's done, the flow is boring and predictable:

git worktree add ../work/<job> -b <job>   # give the agent a sandbox
# ...agent works, commits, opens a PR...
git worktree remove ../work/<job>         # bin the sandbox when merged

Review the branch, merge it, remove the worktree. Three agents, three sandboxes, three clean branches — instead of one contaminated working directory you have to untangle by hand.

Why it beats the obvious alternatives

vs. everyone in one folder, switching branches: no thrashing, no stashing, nobody yanking the rug on an agent mid-edit. Each agent's branch just sits there, untouched, in its own directory.

vs. cloning the repo N times: worktrees share the object store, so they're faster to make and lighter on disk, and every branch is visible from every worktree. Clones drift into separate little islands; worktrees stay one repo.

Parallel tests and builds: each worktree builds on its own. Three agents can run the full test suite at the same time without standing on each other's output.

The honest gotchas — because there are a few

You can't check out the same branch in two worktrees. Git flatly refuses. Sounds annoying; it's actually a gift — it means two agents can't both be editing the same branch by accident.

Cleanup needs discipline. Stale worktrees pile up if you're sloppy. git worktree remove when you're done, and git worktree prune to sweep up dead ones. When an agent dies messily mid-job, you'll occasionally need git worktree remove --force — and yes, that's exactly as fun as it sounds.

Disk and dependencies are real. Each worktree is a full checkout, and things like node_modules or a Python venv don't magically share — so N worktrees on a heavy repo can mean N installs. For most projects it's a non-issue. For a monster monorepo, budget for it.

Something has to manage them. Creating and destroying worktrees is orchestration overhead. Worth every bit of it when you're running agents in parallel; complete overkill if you only ever run one.

The payoff

Three agents, three features, one repo, all at once — each sealed in its own sandbox, none able to touch the others. When they finish, you're merging three clean branches, not performing surgery on a working directory that two agents fought over.

That's the whole point. Worktrees are the filesystem version of the file-ownership rule from last time: the wall that makes "stay in your lane" physically true instead of merely hoped-for. It's dull plumbing. Dull plumbing is usually what separates a demo from something you'd actually trust with real work.
