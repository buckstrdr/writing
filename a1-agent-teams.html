I Run a Team of AI Agents From My Phone. Here's How It Actually Works.

No magic prompt. Just a tmux window, a Discord channel, and a few rules I learned the hard way to stop the whole thing shipping confident rubbish.

Ian Salter · 6 min read

Last week I shipped a feature while standing in a Tesco queue.

I didn't write any of it. I sent one message to a Discord channel — "spin up a builder and a reviewer, do the auth middleware" — and put my phone back in my pocket. By the time I'd paid, three files were written, the tests were green, and a second agent had already moaned that the first one forgot a rate-limit header.

I ran the team. I didn't do the work.

Now, most of what gets written about AI coding is the same picture: one person, one chat window, one agent that either nails it or invents a library that doesn't exist. And that's fine right up until it isn't — usually the moment it matters. The thing that actually changed how I work wasn't a cleverer prompt. It was giving up on the single genius agent and running a small team of ordinary ones instead.

Why one agent isn't enough

A single agent has one problem you can't design around: it marks its own homework. It writes the code, tells you it's great, and sounds exactly as confident when it's wrong as when it's right. It's the developer, the reviewer and QA all at once — which really means it's none of them.

We worked this out for humans decades ago. You don't let the person who wrote the code be the only one who checks it. So I stopped trying to make one agent smarter and started making a few of them argue.

The setup

It all sits in a tmux window — basically a grid of terminals I can see at a glance.

One pane is the boss. It doesn't write code and it doesn't review code. All it does is take my instruction, chop it into jobs, hand them out, and tell me what's happening. The other panes are the workers — each one a fresh agent with a job title (builder, reviewer, researcher) and a model behind it. I deliberately mix the models: a Claude builder, a different model reviewing, so they don't share the same blind spots and nod along with each other.

And they stay in their lanes. The builder builds. The reviewer picks holes. The boss coordinates. Nobody wanders off to "just quickly improve" something I never asked them to touch — which, if you've used these things, you'll know is half the battle.

Running it from my phone

The boss is plugged into a Discord channel. That's the whole remote control.

I message it, it messages back — "builder's on the middleware," "reviewer's flagged a missing test," "done, PR's up." I'm not sat watching a terminal. I'm on the sofa, or in that Tesco queue, and the work reports up to me like a team would.

Sounds like a gimmick. It isn't. The bottleneck in this stuff was never how fast the model types — it's my attention. The second the work started reporting to me in a channel I already check out of habit, I stopped being the thing everything waited on.

The bit nobody mentions: making sure it actually started

This one took me way too long to sort out.

When you tell an agent to do something, you need to know it heard you. Early on I'd fire off a job, assume it landed, and come back twenty minutes later to find the worker never picked it up. The message just… went nowhere.

So now it's not fire-and-forget. When the boss hands out a job, it checks the worker actually started — and gives it a nudge if it didn't. When a worker finishes, it leaves a little "done" marker the boss is watching for. No guessing, no "yeah that's probably finished." A proper handshake at both ends.

Dull as it sounds, that one thing is the difference between a nice demo and something you'd trust with real work.

The rules that keep it honest

The wiring is only half of it. The other half is a handful of rules that all start from the same assumption: the agents will get things wrong, so build for it.

Whoever writes it doesn't review it. Different agent, different model. The reviewer's whole job is to find what the builder missed, and because it didn't write the code, it's got no pride to defend. It'll happily throw the builder under the bus. Good.

Everyone owns their own files. When a few agents work at once, each one gets its own patch and doesn't touch anyone else's. Sounds like admin. It's actually what stops two of them quietly overwriting each other and handing you something that compiles but is subtly broken — the worst kind of broken. (The filesystem version of this — giving every agent its own git worktree so they physically can't tread on each other — is a whole article on its own. That one's coming.)

Make them try to prove it's wrong. For anything that matters I don't ask "is this right?" — I get a few agents to attack it and see if it survives. You'd be amazed how many confident, tidy-looking answers fall apart the moment three agents go at them looking for the crack.

Always say when you're done. Even — especially — when it went wrong. Silence is the one thing this setup can't cope with, because a worker that's quietly stuck looks identical to one that's making progress, right up until the deadline you've now missed.

What it's actually like

Honestly? It feels less like coding and more like running a small, fast, slightly feral team. I spend my time deciding what should happen and reading back what did — not typing it out myself. Some days it's genuinely brilliant. Some days a worker disappears down a rabbit hole and I have to step in like any manager whose bloke's gone quiet on Slack.

And it's not free. Running a few agents at once burns through tokens, and an unsupervised team of them will find whole new ways to be wrong in parallel. Those rules aren't decoration — they're the only reason it isn't just an expensive machine for producing confident nonsense.

But the shift is real, and I don't think it's going back in the box. The leverage was never a smarter agent. It's a better-run team of average ones.

If you fancy trying it

You don't need my exact rig. Start with two agents and one rule: whoever writes it isn't the one who checks it. And give the checker a proper job — "tell me what's wrong with this," not "does this look ok?" That single split will catch more than any amount of prompt-fiddling.

Then, once you trust it, bolt on the boring bits that make it dependable — checked handoffs, clear ownership, a channel that tells you what's going on. None of it's glamorous. It's just the gap between a demo and something you'd actually rely on.

I'm writing up the whole build as I go — the agent setup, the trading systems it grew out of, and the stuff I got wrong first. If that's your kind of thing, follow along.
